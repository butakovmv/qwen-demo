# Соглашения по архитектуре и разработке проекта

1. Стремимся к Low Coupling (низкая связанность) и High Cohesion (высокое зацепление), стараемся ограничивать видимость классов где, это возможно.

1. Контролируем SOLID

1. Придерживаемся гексагональной архитектуры

## Рекомендации по области видимость

| Класс | Видимость | Обоснование |
|---|---|---|
| **Application / main** | `public` (по умолчанию) | Точка входа, должна быть видна |
| **Интерфейсы usecase** (`interface`) | `public` (по умолчанию) | Публичный контракт модуля `usecase` |
| **Модели** (`data class`) | `public` (по умолчанию) | Используются на границах модулей |
| **Реализации usecase** (`@Component class`) | `internal` | Внутренняя деталь, внешний код зависит от интерфейса |
| **Контроллеры** (`@RestController class`) | `internal` | Spring обнаруживает через reflection, извне не вызываются |
| **Конфигурации** (`@Configuration class`) | `internal` | Внутренняя настройка модуля |
| **Параметры конструкторов** | `private val` | Зависимости не должны утекать наружу |
| **Вспомогательные функции/классы** | `private` / `internal` | Минимально необходимая видимость |
